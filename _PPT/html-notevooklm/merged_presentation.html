연구 배경: 고정밀 예측과 "블랙박스" 모델의 모순

Context

  • 에너지 수요 예측(Energy Demand Prediction)은 온실가스 배출 감소와 전력 관리에 매우 중요합니다.

Problem Definition

  • 딥러닝의 딜레마: CNN-LSTM 등의 모델은 정확도가 높지만 설명 가능성(Black-box nature)이 부족합니다.
  • 기존 XAI의 한계: SHAP 및 LIME 등 주요 방법은 인과성(Causality)보다는 특성의 상관관계(Correlation)에 주로 초점을 맞춥니다.
  • 사용자 요구: 최종 사용자에게는 "왜 피크 경고가 발생했는가?"와 같은 실행 가능한 통찰력(Actionable Insights)이 필요합니다.

Research Motivation

  • 딥러닝의 높은 정확도를 유지하면서 베이지안 네트워크를 통해 변수 간 및 모델 파라미터의 인과 메커니즘을 규명하는 하이브리드 프레임워크를 제안합니다.

  • Feature Fusion: 직렬 연결(cascading)이 아닌 CNN 출력과 LSTM Context Vector 병합(concatenation).

딥러닝 파라미터(DLP)의 의미론적 해석

  • Class Activation Map (CAM): K-means 클러스터링을 사용하여 활성화 패턴을 식별하고 기기 사용의 증감을 매핑.
  • Attention Weights: 누적 클러스터링(Cumulative Clustering)을 기반으로 시계열 관심 지점 식별.
  • 의의: 신경망의 "블랙박스" 파라미터를 베이지안 네트워크의 설명 가능한 노드로 변환.

추론 경로 (Inference Path):

왜 병렬 모델 성능이 직렬보다 낮은가?

📊 공정한 비교 실험 결과 (동일 훈련 조건)

모델 검증 MAE 테스트 MAE vs Baseline 논문 예상치
S-CNN-LSTM (baseline) 0.030823 0.031296 - -
S-CNN-LSTM-Att 0.029874 0.029918 ✅ +3.08% -
P-CNN-LSTM-Att (논문) 0.035662 0.035467 ❌ -15.70% ✅ +34.84%

⭐⭐⭐⭐⭐ 가장 유력한 원인 (90%)

비교 베이스라인이 다름
논문은 Kim & Cho (2019)의 구형 방법과 비교했으나, 우리가 구현한 직렬 CNN-LSTM은 이미 최적화된 강력한 베이스라인임.

만약 논문의 베이스라인이 단순 LSTM이었다면 34.84% 향상은 합리적임.

기타 가능한 원인

  • 초매개변수 불일치 (60% 확률) - CNN 필터, LSTM 유닛 등 설정
  • 데이터 전처리 차이 (40% 확률) - 시간 특징 공학, 정규화 세부 사항
  • 아키텍처 세부 사항 차이 (30% 확률) - 가중치 초기화, 활성화 함수
  • 훈련 전략 차이 (30% 확률) - 학습률 스케줄링, 데이터 증강

📌 연구 배경 및 동기

기존의 설명 가능한 에너지 예측 시스템은 주로 "부하 피크가 왜 발생했는가(Why)"에 초점을 맞춥니다. 하지만 실제 에너지 관리 현장에서는 "전력 사용 행동을 바꾸면 어떤 일이 일어나는가(What-if)? 어떤 변화를 취해야 하는가(Which action)?"가 더 중요한 질문입니다.

🎯 핵심 연구 목표

연구 목표를 인과적 설명(Causal Explanation)에서 인과 기반의, 실행 가능한, 사용자 피드백을 통해 지속적으로 진화하는 의사결정 추천 시스템으로 확장합니다.

문제 정식화: 주어진 컨텍스트 조건(계절, 시간, 건물 규모, 현재 부하 상태 등) 하에서, 피크 절감, 에너지 소비 감소, 사용자 수용성 및 안전 제약 간의 최적 균형을 이루는 실행 가능한 개입 방안 세트를 생성 및 추천하고, 실제 사용자 상호작용을 통해 시스템을 더욱 지능적으로 만드는 방법.

🔄 전체 시스템 아키텍처 (6대 모듈)

① 컨텍스트 인식 모듈

환경 정보(계절, 시간, 건물 유형, 현재 부하) 및 모델 내부 상태(DLP, EDP, Attention 유형) 인코딩

② 개입 방안 생성기

전문가 템플릿 라이브러리 기반 후보 방안 집합 인스턴스화, 제어 가능한 변수(에어컨 설정, 장비 제어)만 개입

③ 인과 효과 평가

베이지안 네트워크 do-calculus를 통해 P(Peak|do(π),c) 계산, 상관관계가 아닌 개입 효과 평가

④ Slate Bandit 추천

컨텍스트 인식 방안 정렬, Top-K 목록 추천 선택, 탐색과 활용의 균형

⑤ 강화된 피드백 수집

거부 원인 라벨(너무 번거로움/불편함/업무 지장 등) 및 실행 확인 신호 수집

⑥ 온라인 학습 업데이트

Bandit 전략 업데이트 + 선호도 모델 업데이트 + 생성 전략 진화

  • Slate Bandit + Enhanced Feedback Module: 다목적 보상으로 Top-K 방안 선택, 사용자 선호/거부 원인 결합하여 폐루프 적응.

r(π) = α·AcceptIntent + β·Like - γ·Dislike + λ·ΔPeak + μ·ΔEnergy - ρ·Cost - η·RejectionPenalty

① 거부 원인 분해 (Preference Decomposition)

"부정 피드백"을 스칼라에서 다차원 선호도 라벨로 분해:

RejectReason ∈ {너무 번거로움, 불편함, 업무 지장, 신뢰 불가, 이미 수행 중}
시스템 역할:
• 선호도 모델 훈련 및 구조적 제약 제공
• 다변수 조합 및 조절 범위 제약
② 실행 확인 (Intent vs Execution)

사용자 의향과 실제 실행을 명시적으로 구분:

Accept-Intent 사용자 실행 의향 있음
Executed 실제 실행 확인
왜 구분이 필수적인가?
Executed=1일 때만 실제 부하 변화를 사용하여 인과 효과 평가를 업데이트, 미실행 방안으로 인한 오염 방지
